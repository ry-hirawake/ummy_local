# ADR-0001: AWS前提の認証基盤として Amazon Cognito を採用する

## TL;DR
- Decision: Ummy の認証基盤は Amazon Cognito User Pools を第一選択として採用する
- Flow: Managed Login + OAuth 2.0 Authorization Code Grant with PKCE を使う
- Session: Next.js 側では `httpOnly cookie` ベースのサーバーセッションとして扱う
- Why: AWS本番前提、運用責務の軽さ、OIDC/OAuth標準準拠、Next.js App Router との相性

## Status
Proposed

## Decision
- 認証基盤は Amazon Cognito User Pools を採用する
- ログイン導線は Cognito Managed Login を利用する
- 認可フローは OAuth 2.0 Authorization Code Grant with PKCE を採用する
- Ummy アプリケーションは Cognito のトークンをそのままクライアント配布せず、Next.js サーバー側で検証し `httpOnly cookie` ベースのセッションとして扱う

## Context
- Requirements の `FR-9.1`, `FR-9.3` は認証必須と `httpOnly cookie` 推奨を要求している
- Constraints では AWS 単一ベンダー、本番運用は AWS 前提である
- ユーザーから「認証はAWSに載せることを前提としてベストプラクティスを優先する」という追加方針が出ている
- Next.js App Router ではサーバー側で `cookies()` を扱えるため、クライアントにトークン文字列を持たせない構成が取りやすい
- Amazon Cognito は Authorization Code + PKCE をサポートしており、Managed Login でログインUIと OAuth フローを標準化できる

## Options Considered
### Option A: Amazon Cognito User Pools + Managed Login + PKCE
- Pros:
  - AWS 前提と整合する
  - OIDC / OAuth 2.0 標準フローを利用できる
  - Managed Login によりログイン画面の運用コストを抑えられる
  - PKCE を用いて認可コード傍受リスクを軽減できる
- Cons:
  - Cognito 特有の概念（User Pool, Hosted UI/Managed Login, callback 設定）を理解する必要がある
  - セッション抽象は Next.js 側で実装する必要がある

### Option B: Auth0 など外部SaaS
- Pros:
  - 認証機能が豊富で導入例も多い
  - UI/ID連携の選択肢が広い
- Cons:
  - AWS 単一ベンダー方針とずれる
  - 外部SaaS依存と運用コストが増える
  - 社内専用要件との整合確認が追加で必要

### Option C: 自前認証
- Pros:
  - 挙動を自由に制御できる
  - AWS 上にすべて閉じられる
- Cons:
  - セキュリティ責任と実装負荷が過大
  - OAuth/OIDC の標準実装を自前で安全に維持する必要がある
  - ベストプラクティス優先という条件に反する

## Rationale
- AWS 前提とセキュリティ標準を両立する最短経路が Cognito である
- PKCE は Cognito 公式でも認可コード付与に対する保護として案内されている
- App Router ではサーバー側 cookie 処理がしやすいため、ブラウザ storage へトークンを置かない設計と噛み合う
- Auth0 は実現可能だが、今回の AWS 前提とベンダー方針に対して余分な選択肢である
- 自前認証は保守・監査・セキュリティの観点で不利であり、初期フェーズで採る理由が弱い

## Consequences
- Impacted Specs:
  - story-0005
- Impacted Tests:
  - `src/__tests__/auth-guard.integration.test.tsx`
- Operational concerns:
  - Cognito の callback/logout URL、cookie 属性、セッション期限の管理が必要
  - AWS 環境ごとに User Pool / App Client の設定差分を持つ必要がある

## Migration (if breaking)
- 初期導入のため不要
